# RAG 知识库检索问答 —— 需求文档（暂缓实施，仅记录共识）

> 状态：**未排期**。本文档只是把 2026-07-12 讨论中达成的共识记录下来，防止遗忘。
> 是否真正开发，取决于后续「是否有真实使用需求」的观察结果，见文末「触发条件」。

## 背景

MemoBase 目前的真实使用场景之一：托管一份两年积累的 AI 课程知识库——20 余门课程，每门课 6-12 个
module，包含讲件、笔记及其他资料，合计几百篇文档。这个体量已经超出"翻文件夹能一眼找到"的量级，
产生了"直接对话检索资料"的诉求：用户在搜索入口提问，MemoBase 用内置检索找到相关资料片段，交给
LLM 生成回答（即 RAG：Retrieval-Augmented Generation）。

同一时间点，MemoBase 刚完成 AI Model Provider 配置（`InstanceSetting.AISetting`，见
`proto/api/v1/instance_service.proto` 的 `AIProviderConfig` / `AIModelConfig`），已经有了可复用的
"实例级 LLM 连接"基建，这也是本次讨论 RAG 是否顺手可做的直接动机。

## 讨论结论

### 1）这个体量是否需要 RAG？

需要，但边界要卡清楚：

- 几百篇文档、多门课程/module 的规模，靠人工翻目录或关键词全文搜索已经不够用——问题往往是跨文档、
  跨 module 的（"哪门课讲过 fine-tuning vs RAG 的取舍"），这正是 RAG 要解决的场景。
- 但**不需要对齐工业级方案**。不必引入 Pinecone/Weaviate 等外部向量数据库，本项目当前体量下，
  `sqlite-vec`（或直接在 SQLite 里存 embedding 做暴力余弦相似度）完全够用，且不引入新基础设施，
  与项目"单 Go 二进制、SQLite 优先"的既有取向一致。

### 2）现成的第三方 RAG 产品能替代吗？

可以部分替代，但不完全等价：

- 路径是：用 `memogit`（规划中）把知识库检出到本地文件夹 → 喂给 Cherry Studio 等现成 RAG 客户端做
  检索问答。这条路径**可达性更弱**（多一次导出/同步动作，数据新鲜度依赖检出频率），但**开发成本为
  零**——如果只是"我自己能用起来"，这条路径已经够用，不构成"必须给 MemoBase 内置 RAG"的理由。
- 内置的价值增量在于：MemoBase 直接提供"搜索 + 聊天"入口，数据永远最新（无需导出同步），且能利用
  MemoBase 已有的结构化元数据（workspace / folder_path）做范围过滤——这是通用 RAG 客户端不具备的、
  只有"数据源自己做检索"才有的优势。

### 3）值不值得开发？

分两层结论：

- **作为面向他人的产品功能：价值有限。** MemoBase 明确不以推广、涨 star、吸引 fork 为目标，
  "会有更多人需要"这个前提目前不成立，不构成开发的驱动力。
- **作为一次真实的 RAG 实践：值得，且是本次开发的唯一目的。** 用户认可这是难得的"真实数据 + 真实
  使用动机 + 边界明确"的练手机会，比公开数据集练习更有参考价值，因此明确愿意投入去做，但**目的是
  积累 RAG 实操经验，不是为了打磨 MemoBase 这个产品**。这意味着实现应该以"验证核心检索质量"为
  重点，不必做多租户隔离、权限体系等面向陌生用户的防御性设计。

### 4）触发条件（何时启动排期）

本文档记录的是共识，不是待办。仅当出现以下情况之一时，再回来规划排期：

1. 用户在真实使用中反复遇到"翻目录/关键词搜索找不到想要的资料"的痛点，验证了检索问答确有日常
   使用价值；或
2. 用户主动决定投入时间做一次 RAG 实操练习，无论是否有强烈使用痛点驱动。

在此之前，不进入开发计划，也不预先做基建（如提前建 embedding 表）。

## 范围（若启动，最小可行范围）

若后续决定启动，第一版应聚焦验证检索质量本身，而非完整产品化：

**要做：**

- 对现有文档做 chunking + embedding，存入 SQLite（`sqlite-vec` 或等价的向量列 + 暴力余弦相似度）。
- 复用已有的 `AIProviderConfig` 做 embedding 调用与生成式回答调用，不新增一套独立的 provider 配置。
- 一个搜索/聊天入口：用户提问 → 检索相关 chunk → 连同问题一起交给 LLM 生成回答，回答附来源文档链接。
- 用真实会问的 10-20 个问题做人工评估检索质量（chunk 是否命中预期文档），而不是一上来就打磨聊天 UI。

**明确不做：**

- 多租户隔离、按用户维度的权限收敛（因为不面向陌生用户）。
- 外部向量数据库、独立检索服务等新增基础设施。
- 承诺"生产级"检索精度或复杂的 rerank/hybrid search 管线——先用最简单的方案验证够不够用，
  不够用再迭代。

具体的 chunking 策略、embedding 存储位置、检索 API 形态见配套的技术方案文档
